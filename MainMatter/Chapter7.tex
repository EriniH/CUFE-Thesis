\section{Introduction}
\label{sec:pp-introduction}
The post-processing framework is the analysis and visualization layer developed to process and interpret circuit simulation results. After the simulation phase, the generated waveform data must be converted into meaningful engineering information through measurements, mathematical operations, and graphical representation.

The developed framework provides a unified command-driven environment that enables users to interact directly with simulation results through a programmable interface. It integrates waveform management, numerical measurement extraction, mathematical expression evaluation, and automated visualization within a single workflow.

The framework is based on an in-memory Waveform Database that stores and organizes simulation results after parsing standard SPICE raw files. This database provides a common data representation that allows different processing modules, including the Measurement Engine, Calculation Engine, and Visualization Engine, to operate efficiently on the same waveform data.

For visualization, the framework utilizes Gnuplot as an external rendering backend. The Plot Engine is responsible for preparing waveform data and generating the required plotting instructions, while Gnuplot performs the final graphical rendering. This separation provides a flexible visualization architecture while keeping the analysis modules independent from the plotting implementation.

This chapter presents the design and implementation of the proposed post-processing framework. It describes the system architecture, waveform data model, command interface, processing algorithms, visualization pipeline, and validation scenarios used to evaluate the accuracy, robustness, and reliability of the developed tool.

% ---------------------------------------------------------------------------
The framework consists of the following components:

\begin{enumerate}
\item \textbf{RawfileReader} --- Parses the standard SPICE raw file format by extracting metadata from the ASCII header and reconstructing waveform samples from the binary data section.
\item \textbf{WaveformDB} --- Maintains loaded simulation records and provides unified, analysis-type-aware signal lookup.
\item \textbf{Interpreter} --- Tokenises user input and dispatches commands to the appropriate analysis engines.
\item \textbf{MeasEngine} --- Computes numerical characteristics such as maximum value, rise time, frequency, and delay from waveform data.
\item \textbf{CalcEngine} --- Evaluates mathematical expressions over signals and stores derived waveforms for later analysis.
\item \textbf{PlotEngine} --- Generates engineering waveform visualization using \texttt{gnuplot}, including automatic Bode plot generation for AC simulation results.
\end{enumerate}

Overall, the architecture creates a clear separation between simulation data acquisition, waveform management, user interaction, and analysis operations. This modular structure enables reliable extension of the tool with additional measurements, commands, or visualization capabilities.

% ---------------------------------------------------------------------------


\paragraph{Architectural Design Patterns and Trade-offs.}

The post-processing architecture follows stringent software engineering and methodological design principles to balance performance with extensibility:

\begin{itemize}
\item \textbf{Separation of Concerns (Decoupled Pipeline):} The framework strictly isolates file parsing, data storage, command dispatching, and numerical analysis. This allows the visualization backend or the parsing format to be swapped entirely without affecting the core mathematical engines.
\item \textbf{Command Pattern for Extensibility:} The user interface operates on the \texttt{ICommand} abstraction. This design pattern ensures that integrating a new analysis feature does not require modifying the central interpreter logic, thus adhering to the Open/Closed Principle.
\item \textbf{In-Memory Caching (Data Reuse):} The \texttt{WaveformDB} acts as an in-memory caching layer for processed simulation results. This introduces a space-time trade-off: it increases peak memory utilization but eliminates repetitive disk I/O, dramatically accelerating sequential analysis commands.
\item \textbf{Stateless Engines (Analysis Independence):} The Measurement, Calculation, and Plotting engines are implemented as stateless processors that operate purely on data vectors provided by the \texttt{WaveformDB}. This lack of internal state minimizes side effects and ensures deterministic evaluation of expressions.
\end{itemize}

\section{Data Model and Numerical Formulation}
\label{sec:pp-datamodel-algorithms}
% ---------------------------------------------------------------------------
\subsection{Waveform Representation}

A simulation result is represented as a discrete waveform:

\begin{equation}
W = \{(x_i,y_i)\}_{i=0}^{N-1}
\end{equation}

where $x_i$ represents the independent variable and $y_i$ represents the corresponding waveform sample. For AC analysis, each sample is represented internally using its real and imaginary components.

The independent variable may represent time in transient analysis, frequency in AC analysis, or a swept parameter in DC analysis.

\subsection{\texttt{RawfileReader}}
\label{subsec:rawfile-reader}

The \texttt{RawfileReader} is implemented as a stateless utility component that parses a simulation output file and returns a fully populated in-memory structure.  The file format follows standard SPICE output conventions: an ASCII header section describes the metadata and variable list, followed by a binary data block that stores all sample values in row-major order.

\paragraph{Parsing Algorithm}

Algorithm~\ref{alg:raw-reader} summarizes the main parsing procedure.

\begin{algorithm}[H]
\caption{ReadRawFile}
\label{alg:raw-reader}
\textbf{Input:} binary \texttt{.raw} file \\
1. Read ASCII header \\
2. Extract simulation metadata \\
3. Detect analysis type \\
4. Allocate waveform vectors \\
5. Parse binary block \\
6. Convert samples \\
7. Return \texttt{RawFile} object
\end{algorithm}

The parsing process consists of two main stages: metadata extraction and binary waveform reconstruction.
\newpage

\paragraph{Header fields parsed.}
The reader extracts the following fields from the ASCII header:
\begin{itemize}
  \item \texttt{Title} --- a free-text description of the simulation.
  \item \texttt{Date} --- the timestamp written by the simulator.
  \item \texttt{Plotname} --- a keyword that identifies the analysis type (\texttt{Transient}, \texttt{AC}, \texttt{DC}, or \texttt{Operating Point}).
  \item \texttt{Flags} --- set to \texttt{complex} for AC analyses; absent or empty for real analyses.
  \item \texttt{No.\ Variables} --- total number of signals (including the sweep variable).
  \item \texttt{No.\ Points} --- total number of time/frequency steps.
  \item \texttt{Variables} block --- one tab-delimited line per signal giving its index, name, and type.
\end{itemize}

\paragraph{Binary data layout and Constraints.}
For \emph{real} analyses the binary section stores doubles in row-major order. For \emph{complex} AC analyses the sweep variable occupies a single double per row, while each remaining variable occupies two consecutive doubles (real part followed by imaginary part). The reader reconstructs the waveform vectors by mapping each binary sample position to the corresponding variable index extracted from the header. Only the \texttt{Binary:} data section is supported. Signal names are looked up case-insensitively.

\subsection{Waveform Database}
\label{subsec:waveform-db}

The waveform database (\texttt{WaveformDB}) is the central in-memory repository that maintains all loaded simulation results. The database abstraction avoids direct coupling between file parsing and analysis modules. After loading, all engines operate on a unified waveform representation rather than depending on the original storage format. Each call to the \texttt{load} method parses one raw file and appends a \emph{plot record} to an internal list.

The database also supports derived signals created by the expression calculator. When a \texttt{calc} command produces a waveform result, that waveform is stored in an overlay map and subsequently treated as a first-class signal by measurement and plotting commands. Signal lookup is therefore performed in the following order:
\begin{enumerate}
  \item Computed signals created by \texttt{calc}.
  \item Raw records in reverse-load order (most recently loaded first).
  \item Lazy-evaluated complex accessors such as \texttt{mag(...)}, \texttt{ph(...)}, \texttt{db(...)}, \texttt{re(...)}, and \texttt{im(...)}.
\end{enumerate}
This lookup order enables convenience functions such as \texttt{plot freq\_Hz V(2)} while preserving explicit user control over derived expressions.

\subsection{PlotRecord}
\label{subsec:plotrecord}
Each record carries:
\begin{itemize}
  \item A short \emph{tag} derived automatically from the file stem (e.g., \texttt{test\_rc\_TRAN\_0}).
  \item The analysis type: \texttt{OP}, \texttt{DC}, \texttt{TRAN}, or \texttt{AC}.
  \item The X-axis vector and a map of Y signal names to sample vectors.
  \item For AC results, the imaginary column is named with a \texttt{\_im} suffix.
\end{itemize}

\paragraph{Signal look-up order.}
When a command requests a signal by name, the database searches:
\begin{enumerate}
  \item Computed (derived) signals created by the \texttt{calc} command.
  \item Raw records in reverse-load order (most recently loaded first).
  \item Lazy-evaluated complex-accessor wrappers (\texttt{mag(\ldots)}, \texttt{ph(\ldots)}, \texttt{db(\ldots)}, \texttt{re(\ldots)}, \texttt{im(\ldots)}).
\end{enumerate}

\section{Command Interface}
\label{sec:command-interface}

\subsection{Grammar}
\label{subsec:command-grammar}

The command-line interface follows a small verb-first grammar. The syntax is intentionally constrained so that every command can be parsed deterministically before it is dispatched to a specialised engine.

\begin{itemize}
  \item \textbf{Command Structure:} \texttt{<Verb> <Arguments> <Options>}
  \item \textbf{Plot Command:} \texttt{plot <XAxis> <YSignal>+ <PlotOptions>}
  \item \textbf{Measurement Command:} \texttt{meas <MeasureType> <Signal> [<MeasureArgs>]}
  \item \textbf{Calculation Command:} \texttt{calc <Expression>}
\end{itemize}

The interpreter tokenises input on whitespace and commas while preserving quoted strings, then dispatches each command to a concrete \texttt{ICommand} handler object. This architecture keeps parsing, validation, and execution separate, which simplifies extension and reduces the chance of inconsistent error handling.

% ---------------------------------------------------------------------------
\subsection{Execution Modes}
\label{sec:pp-invocation}
% ---------------------------------------------------------------------------

The post-processor is invoked as a standalone executable.  Three modes of operation are supported:

\begin{description}
  \item[One-shot mode] One or more raw files are given as positional arguments followed by a command and its arguments.  The tool loads all files, executes the single command, prints the result, and exits.  Example syntax: \\[0.3em]
   \textit{file.raw} [\textit{file2.raw}~\ldots] \textit{command} [\textit{args}]

  \item[Interactive mode] Invoked with the \texttt{-i} (or \texttt{--interactive}) flag, or with no arguments at all, the tool starts a prompt-driven REPL (\texttt{spice>}).  The user may load additional files, run any number of commands, and exit by typing \texttt{exit}, \texttt{quit}, or \texttt{q}.

  \item[Script mode] Invoked with \texttt{--script \textit{file}}, the tool reads commands from a plain-text script file line by line.  Lines beginning with \texttt{\#} are treated as comments and skipped.
\end{description}

In all modes, netlist files with extensions \texttt{.cir}, \texttt{.sp}, or \texttt{.prm} supplied on the command line are automatically simulated before any command is executed, and their results are loaded into the database. The \texttt{source} command also provides an explicit simulator interface for running netlist files in-process with options such as \texttt{-v}/\texttt{--verbose}, \texttt{-o}/\texttt{--output-dir}, \texttt{--raw}, and \texttt{--no-raw}.

Among the available commands, the core analytical and visualization capabilities are represented by the \texttt{meas}, \texttt{calc}, and \texttt{plot} commands. These commands are detailed below.

\subsubsection{\texttt{meas}}
\textbf{Syntax:} \texttt{meas \textit{mtype} \textit{sig} [\textit{params}]}\\
\textbf{Example:} \texttt{spice> meas max V(out) from 1ms to 2ms}\\
\textbf{Description:} Computes a scalar measurement on the named signal. Supported measurement types and their parameters are described in Section~\ref{sec:pp-meas}.

\subsubsection{\texttt{calc}}
\textbf{Syntax:} \texttt{calc \textit{expression}}\\
\textbf{Example:} \texttt{spice> calc V(1) * I(R1)}\\
\textbf{Description:} Evaluates an arithmetic expression over signals. Waveform results are stored in the database under the expression string, making them available for subsequent commands.

\subsubsection{\texttt{plot}}
\textbf{Syntax:} \texttt{plot \textit{xsig} \textit{ysig} [\ldots] [--out \textit{file}] [--title \textit{"title"}]}\\
\textbf{Example:} \texttt{spice> plot time V(out) --out transient.png --title "Output Voltage"}\\
\textbf{Description:} Produces a PNG waveform plot (time-domain, DC, or Bode for AC). The plot engine automatically selects the appropriate rendering mode based on the X-axis signal name.

\subsubsection{\texttt{source}}
\textbf{Syntax:} \texttt{source \textit{netlist.cir} [ -v | --verbose ] [ -o DIR | --output-dir DIR ] [ --raw | --no-raw ]}\\
\textbf{Example:} \texttt{spice> source amplifier.cir -v --output-dir ./results}\\
\textbf{Description:} Runs the circuit simulator in-process on the given netlist file. The \texttt{-v/--verbose} flag enables detailed diagnostic output. The \texttt{-o/--output-dir} specifies where to write result files. The \texttt{--raw} (default) writes binary \texttt{.raw} files; \texttt{--no-raw} suppresses binary output and prints simulation results to the console/CSV. After simulation, the framework automatically loads any generated \texttt{.raw} files into the database.

\subsubsection{\texttt{export}}
\textbf{Syntax:} \texttt{export --csv \textit{outfile} [\textit{signal1}] [\textit{signal2}] ...}\\
\textbf{Example:} \texttt{spice> export --csv data.csv time V(out) V(in)}\\
\textbf{Description:} Exports the specified signals (or all signals if none are given) to a CSV file. The X-axis (e.g., time) is included if not explicitly listed. The output is written to a subdirectory \texttt{exports/} next to the raw file, unless an absolute path is provided.

\subsubsection{Shell Utilities}
The following commands provide basic file-system navigation within the REPL:

\begin{itemize}
  \item \texttt{cd [path]} --- changes the current working directory. If no path is given, changes to the user's home directory.
  \item \texttt{ls [path]} --- lists the contents of the specified directory (or the current directory if no path is given). Directories are shown with a trailing slash.
  \item \texttt{pwd} --- prints the absolute path of the current working directory.
\end{itemize}

\subsection{Signal Convention and Examples}

Signal names follow the convention established by the simulator:
\begin{itemize}
  \item \textbf{Node voltages:} \texttt{V(out)}, \texttt{V(1)}, \texttt{V(vdd)}
  \item \textbf{Branch currents:} \texttt{I(R1)}, \texttt{Ib(Q1)}
  \item \textbf{Sweep variable:} \texttt{time} (transient), \texttt{freq\_Hz} (AC)
  \item \textbf{AC imaginary part:} \texttt{V(2)\_im} (automatically generated)
  \item \textbf{Complex accessors:} \texttt{mag(sig)}, \texttt{ph(sig)}, \texttt{db(sig)}, \texttt{re(sig)}, \texttt{im(sig)} --- these are evaluated lazily and do not require explicit storage.
\end{itemize}
Signal names are matched case-insensitively.

\section{Processing Algorithms}

\subsection{Measurement Algorithms}
\label{sec:pp-meas}
% ---------------------------------------------------------------------------

The measurement module, implemented by the \texttt{MeasEngine} class, computes scalar metrics from waveform data. All measurements return a numeric result with an inferred unit (e.g., \si{\volt}, \si{\ampere}, \si{\second}, \si{\hertz}).

Measurements like \texttt{max}, \texttt{min}, \texttt{ptp}, \texttt{avg}, and \texttt{rms} accept an optional \texttt{from}~\textit{t1} \texttt{to}~\textit{t2} suffix that restricts the window of data over the X-axis.

The available measurements are classified into the following categories.

\subsubsection{Statistical Measurements}

\paragraph{\texttt{max}}
\textbf{Syntax:} \texttt{meas max \textit{sig} [from \textit{t1} to \textit{t2}]}\\
\textbf{Example:} \texttt{spice> meas max V(out) from 1ms to 2ms}\\
\textbf{Description:} Computes the maximum sample value of the signal within the optionally specified window.

\paragraph{\texttt{min}}
\textbf{Syntax:} \texttt{meas min \textit{sig} [from \textit{t1} to \textit{t2}]}\\
\textbf{Example:} \texttt{spice> meas min V(out)}\\
\textbf{Description:} Computes the minimum sample value of the signal within the optionally specified window.

\paragraph{\texttt{ptp}}
\textbf{Syntax:} \texttt{meas ptp \textit{sig} [from \textit{t1} to \textit{t2}]}\\
\textbf{Example:} \texttt{spice> meas ptp V(out)}\\
\textbf{Description:} Computes the peak-to-peak swing ($\max - \min$) of the signal.

\paragraph{\texttt{avg}}
\textbf{Syntax:} \texttt{meas avg \textit{sig} [from \textit{t1} to \textit{t2}]}\\
\textbf{Example:} \texttt{spice> meas avg V(out)}\\
\textbf{Description:} Computes the arithmetic mean of the signal.

\paragraph{\texttt{rms}}
\textbf{Syntax:} \texttt{meas rms \textit{sig} [from \textit{t1} to \textit{t2}]}\\
\textbf{Example:} \texttt{spice> meas rms I(R1)}\\
\textbf{Description:} Computes the root mean square of the signal.

\subsubsection{Timing Measurements}

\paragraph{\texttt{cross}}
\textbf{Syntax:} \texttt{meas cross \textit{sig} [\textit{level}] [rise|fall]}\\
\textbf{Example:} \texttt{spice> meas cross V(out) 1.5 rise}\\
\textbf{Description:} Finds the time of the first threshold crossing at the specified level.

\paragraph{\texttt{trise}}
\textbf{Syntax:} \texttt{meas trise \textit{sig} [\textit{lo} \textit{hi}]}\\
\textbf{Example:} \texttt{spice> meas trise V(out) 0.1 0.9}\\
\textbf{Description:} Computes the rise time between fractional or percentage thresholds (e.g., 10\% to 90\%).

\paragraph{\texttt{tfall}}
\textbf{Syntax:} \texttt{meas tfall \textit{sig} [\textit{lo} \textit{hi}]}\\
\textbf{Example:} \texttt{spice> meas tfall V(out)}\\
\textbf{Description:} Computes the fall time between fractional or percentage thresholds.

\paragraph{\texttt{delay}}
\textbf{Syntax:} \texttt{meas delay \textit{sig\_in} \textit{sig\_out}}\\
\textbf{Example:} \texttt{spice> meas delay V(in) V(out)}\\
\textbf{Description:} Computes the time difference between the 50\% crossings of the input and output signals.

\subsubsection{Frequency Measurements}

\paragraph{\texttt{period}}
\textbf{Syntax:} \texttt{meas period \textit{sig} [\textit{level}]}\\
\textbf{Example:} \texttt{spice> meas period V(osc)}\\
\textbf{Description:} Computes the time between two consecutive rising crossings of the specified level.

\paragraph{\texttt{freq}}
\textbf{Syntax:} \texttt{meas freq \textit{sig} [\textit{level}]}\\
\textbf{Example:} \texttt{spice> meas freq V(osc)}\\
\textbf{Description:} Computes the reciprocal of the period, yielding the frequency in \si{\hertz}.

\subsubsection{Other Measurements}

\paragraph{\texttt{find}}
\textbf{Syntax:} \texttt{meas find \textit{sig\_y} when \textit{sig\_x}=\textit{val} [cross=\textit{N}]}\\
\textbf{Example:} \texttt{spice> meas find I(R1) when V(out)=2.5 cross=2}\\
\textbf{Description:} Interpolates \textit{sig\_y} at the exact point where \textit{sig\_x} reaches \textit{val}. The optional \texttt{cross=N} parameter selects the $N$-th crossing if the condition occurs multiple times.

\subsection{Measurement Specification and User Interaction}
\label{sec:meas-spec}

The measurement interface maps a user command directly to a scalar numeric result and an inferred physical unit. The module handles various scenarios automatically:

\begin{itemize}
  \item \textbf{Global \& Windowed Search:} Computes metrics over the entire record or restricts the computation to a specific range using \texttt{from} and \texttt{to} keywords (e.g., \texttt{meas max V(2) from 0 to 1ms}).
  \item \textbf{Thresholds \& Crossings:} Computes rise times and multi-signal delays by interpolating crossing times at defined thresholds.

  \begin{algorithm}[H]
  \caption{Rise Time Measurement}
  \textbf{Input:} waveform $W$, low threshold $L$, high threshold $H$ \\
  1. Find first crossing of $L$ \\
  2. Interpolate crossing time $t_L$ \\
  3. Find first crossing of $H$ \\
  4. Interpolate crossing time $t_H$ \\
  5. Return $t_H - t_L$
  \end{algorithm}

  \item \textbf{Value Interpolation:} Finds precise values at specific X-axis coordinates using linear interpolation.
  \item \textbf{Validation \& Constraints:} Validates periodic constraints (e.g., requiring at least two crossings for \texttt{freq}) and rejects mismatched analysis types. Unsupported commands trigger fuzzy-matched suggestions.
\end{itemize}



\subsection{Numerical Formulation and Interpolation}
\label{sec:pp-mathmodels}

The post-processing framework evaluates analytical models on discrete, sampled waveform data. Because SPICE simulators utilize variable timestep integration methods, the resulting waveform vectors are non-uniformly sampled. Consequently, analytical operations must account for interpolation errors and discrete sampling bounds. The mathematical definitions for the primary operations are as follows:

\paragraph{Global Maximum}
\[
y_{\max} = \max(y_0, y_1, \dots, y_{N-1})
\]

\paragraph{Peak-to-Peak Swing}
\[
V_{pp} = V_{\max} - V_{\min}
\]

\paragraph{Average Value}
\[
V_{avg} = \frac{1}{N} \sum_{i=0}^{N-1} y_i
\]

\paragraph{Signal Delay}
\[
t_d = t_{50,\text{out}} - t_{50,\text{in}}
\]

\paragraph{Rise Time}
\[
t_r = t_{H} - t_{L}
\]
where $t_H$ and $t_L$ are the interpolated crossing times at the high and low thresholds, respectively.

\paragraph{Frequency}
\[
f = \frac{1}{T}
\]

\paragraph{Root Mean Square (RMS)}
\[
RMS = \sqrt{\frac{1}{N} \sum_{i=0}^{N-1} y_i^2}
\]
where $y_i$ is the $i$-th waveform sample.

\paragraph{Linear Interpolation}
\[
y = y_1 + \frac{x - x_1}{x_2 - x_1} (y_2 - y_1)
\]

Binary operations follow broadcast semantics:
\begin{itemize}
  \item scalar $\circ$ scalar $\rightarrow$ scalar
  \item scalar $\circ$ waveform $\rightarrow$ waveform (scalar broadcast)
  \item waveform $\circ$ waveform $\rightarrow$ waveform (element-wise; lengths must match)
\end{itemize}

\paragraph{Built-in Functions.}
The calculator supports the following element-wise functions on waveforms:
\texttt{abs}, \texttt{neg}, \texttt{sqrt}, \texttt{ln}, \texttt{log},
\texttt{db}, \texttt{exp}, \texttt{sin}, \texttt{cos}, \texttt{tan}.

It also supports reduction functions that convert a waveform to a scalar:
\texttt{max}, \texttt{min}, \texttt{avg}, \texttt{mean}, \texttt{rms}.

If an evaluated result is a waveform, it is automatically stored in the database under the text of its expression (e.g. \texttt{V(out)-V(in)}), making it available to subsequent \texttt{plot} and \texttt{meas} commands.

\subsection{Expression Evaluation Specification and User Interaction}
\label{sec:calc-spec}

The expression calculator accepts arithmetic expressions defined by the supported grammar and produces either a scalar or a waveform result. The subsections below specify the expected output for representative expressions drawn from the automated test suite.

\subsubsection{Scalar Arithmetic}
\textbf{Syntax/Example:} \texttt{calc 2+3*4}\\
\textbf{Description:} A valid scalar expression is supplied. The recursive descent parser applies standard operator precedence: multiplication before addition. Scalar result \texttt{14.0} is printed. No signal is stored because the result is a pure scalar.

\subsubsection{Engineering Suffixes}
\textbf{Syntax/Example:} \texttt{calc 10n}\\
\textbf{Description:} The literal uses a recognised engineering suffix. The parser recognises the engineering suffix \texttt{n} and scales the literal by $10^{-9}$. Scalar result \texttt{1e-8} is printed. Other supported suffixes include \texttt{f}, \texttt{p}, \texttt{u}, \texttt{m}, \texttt{k}, \texttt{M}, and \texttt{G}.

\subsubsection{Waveform Arithmetic}
\textbf{Syntax/Example:} \texttt{calc V(1)-V(2)}\\
\textbf{Description:} Both signals are loaded and have matching lengths. The module retrieves both signal vectors from the database and performs element-wise subtraction. A waveform result is produced and stored in the database under the key \texttt{V(1)-V(2)}, making it available to \texttt{plot} and \texttt{meas}.

\subsubsection{Element-wise Functions}
\textbf{Syntax/Example:} \texttt{calc abs(V(2))}\\
\textbf{Description:} A waveform signal is available for transformation. The \texttt{abs} function is applied element-wise to every sample of \texttt{V(2)}. A waveform whose samples equal $|\texttt{V(2)}[i]|$ is stored in the database.

\subsubsection{Reduction Functions}
\textbf{Syntax/Example:} \texttt{calc max(V(2))}\\
\textbf{Description:} The reduction argument resolves to a loaded waveform. The \texttt{max} reduction function collapses the waveform to a single scalar equal to the global maximum sample. A scalar result is printed. No waveform is stored because the result is not a vector.

\subsubsection{Division by Zero}
\textbf{Syntax/Example:} \texttt{calc 1/0}\\
\textbf{Description:} The expression reaches a division-by-zero case. Division by zero is detected during scalar arithmetic evaluation. \texttt{CalcResult.valid} is set to \texttt{false} and an error message is returned. The database is not modified.

\subsubsection{Missing Operand Error}
\textbf{Syntax/Example:} \texttt{calc V(1)+}\\
\textbf{Description:} The parser reaches end-of-input while expecting a right-hand operand. A parse error is reported. The incomplete expression is rejected and the database is left unchanged.

\subsubsection{Unmatched Parenthesis Error}
\textbf{Syntax/Example:} \texttt{calc (2+3}\\
\textbf{Description:} The expression contains an unmatched parenthesis. The parser detects an unclosed parenthesis. A syntax error is returned. The interactive REPL prints the error and awaits the next command without crashing.

\subsection{Plot Generation}
\label{sec:pp-plot}
% ---------------------------------------------------------------------------

The visualization module, implemented by the \texttt{PlotEngine} class, converts waveform data into graphical PNG output using \texttt{gnuplot}. PlotEngine does not directly generate graphical primitives. Instead, it converts waveform data into an intermediate format and delegates rendering to \texttt{gnuplot}. This separation allows the visualization backend to be replaced without modifying analysis logic. It dynamically writes a temporary CSV data file and a \texttt{gnuplot} script before executing \texttt{gnuplot} as a subprocess.
\newpage

\paragraph{Plot Generation Algorithm}

Algorithm~\ref{alg:plot-engine} summarizes the visualization workflow.

\begin{algorithm}[H]
\caption{GeneratePlot}
\label{alg:plot-engine}
\textbf{Input:} X-axis signal, Y-axis signals, plot options\\

1. Validate requested signals\\
2. Detect analysis type\\
3. Select rendering mode (normal/Bode)\\
4. Generate temporary data file\\
5. Generate gnuplot configuration\\
6. Execute gnuplot backend\\
7. Validate generated output\\
8. Return plot result
\end{algorithm}

\paragraph{Rendering Modes.}
\begin{description}
  \item[Normal Mode:] Time-domain or DC signals. Uses linear axes. Multiple signals are grouped into dynamic sub-panels based on their data ranges and physical types (e.g. current vs. voltage) to avoid squashing small signals.
  \item[Bode Mode:] Triggered automatically if the X-axis is \texttt{freq\_Hz}. Generates two stacked panels (Magnitude in dB, Phase in degrees) using a logarithmic frequency X-axis. Automatically utilizes the \texttt{\_im} complex columns.
\end{description}

\paragraph{ASCII Fallback.}
The module searches for \texttt{gnuplot} locally or on the system PATH. If \texttt{gnuplot} cannot be found, it gracefully falls back to rendering a simplified ASCII representation of the plot directly into the terminal console.

\paragraph{Output Path Handling.}
Plots are saved in a \texttt{plots/} directory situated next to the raw file. Names are sanitised by substituting underscores for spaces and parentheses.

\subsection{Error Handling}
\label{sec:pp-errors}
% ---------------------------------------------------------------------------

To ensure robustness and user-friendliness, the Post-Processing module implements a structured error handling strategy across its architectural layers. Rather than allowing the application to crash or exit unexpectedly, errors are carefully intercepted, formatted, and presented to the user, allowing the interactive REPL to safely await the next command.

\subsubsection{Error Handling Strategy}

The error propagation mechanism relies on a hybrid approach of C++ exceptions for fatal I/O operations and state-based return structs for domain logic. This separation is deliberate: exceptions are reserved for unexpected system failures such as missing or malformed input files, while expected computational failures are represented explicitly through return values. This ensures predictable control flow and avoids the performance overhead of stack unwinding during analysis operations, as illustrated in Figure~\ref{fig:error_handling_strategy}.

File I/O operations in \texttt{RawfileReader} throw \texttt{std::runtime\_error} when a file cannot be opened or the header is invalid. The interpreter catches these exceptions during \texttt{load} and \texttt{source} execution, prints a formatted error message, and continues without terminating the REPL.

The post-processing framework is designed to recover cleanly from invalid user input without terminating the REPL. It separates fatal file errors from expected domain failures and reports both through consistent, user-friendly messages.

File I/O errors are handled by exceptions because missing or corrupted raw files are true failures at the system boundary. Analytical failures inside the engines are represented by result structs rather than exceptions.

The interpreter is the central coordinator for error handling. It validates input, catches I/O exceptions, evaluates engine results, and formats all failures with an \texttt{[ERROR]} prefix.

The interpreter uses a consistent workflow for both I/O and domain errors:
\begin{enumerate}
    \item \textbf{Validate the command:} check syntax and required parameters before execution.
    \item \textbf{Catch I/O exceptions:} file loading errors from \texttt{RawfileReader} are caught and reported as user-facing errors.
    \item \textbf{Inspect engine results:} \texttt{CalcResult} and \texttt{MeasResult} carry a boolean \texttt{valid} flag and an \texttt{error} string.
    \item \textbf{Reject invalid results:} failed calculations or measurements do not alter the database.
    \item \textbf{Suggest corrections:} unknown signals and commands trigger fuzzy-match suggestions and help text.
\end{enumerate}

\subsubsection{Error Classification and Examples}

Errors are categorised into four main types, each handled according to its nature:

\begin{enumerate}
  \item \textbf{File I/O Errors:} Occur during raw file parsing. They are thrown as exceptions and caught at the interpreter level. Examples: file not found, malformed header, unsupported ASCII data format.
  \item \textbf{Command Syntax Errors:} Arise from incomplete or incorrectly formatted commands. The interpreter detects these before dispatching and reports a usage hint.
  \item \textbf{Domain Errors:} Originate from mathematical or logical failures inside the engines, such as division by zero, logarithm of a negative number, or insufficient data for period detection. These are returned as part of a result struct and printed with contextual information.
  \item \textbf{Data Availability Errors:} Occur when a requested signal or record does not exist, or when the analysis type constraint does not match any loaded record. The interpreter uses fuzzy matching to suggest close matches.
\end{enumerate}

\section{Validation and Evaluation Scenarios}
\label{sec:pp-verification}
% ---------------------------------------------------------------------------

To demonstrate the robustness, mathematical correctness, and operational stability of the developed framework, a comprehensive empirical evaluation was conducted. The evaluation methodology comprises hundreds of automated scenarios testing the limits of the measurement, mathematical, and visualization engines. These validation scenarios emulate real-world engineering workflows and stress-test the framework's numerical stability against edge cases.

\subsection{Scenario A: Numerical Accuracy and Boundary Handling}
\label{sec:meas-verify}

The first validation scenario assesses the accuracy of the \texttt{MeasEngine} and its ability to handle constraints:

\begin{itemize}
    \item \textbf{Scalar Correctness against Analytical References:} Standard statistical measurements (\texttt{max}, \texttt{min}, \texttt{ptp}, \texttt{avg}, \texttt{rms}) computed by the engine were benchmarked against reference values derived independently from raw signal vectors using rigorously validated standard library algorithms. The framework successfully matched all expected outputs with an error margin within $\pm 10^{-9}$.
    \item \textbf{Windowed Slicing Fidelity:} The capability to restrict computations to an explicit X-axis window (\texttt{from}~$t_1$ \texttt{to}~$t_2$) was evaluated. The framework correctly sliced non-uniformly sampled vectors and accurately handled out-of-bounds requests (e.g., requesting a window beyond the simulation duration) by gracefully returning domain errors without memory faults.
    \item \textbf{Time-Domain Interpolation Integrity:} Transient analyses commands such as \texttt{trise}, \texttt{cross}, \texttt{delay}, and \texttt{find} require accurate linear interpolation between discrete time steps. The evaluation confirmed that the engine accurately identifies multi-level threshold crossings and computes values precisely, effectively avoiding catastrophic cancellations or \texttt{NaN} propagation.
\end{itemize}

\subsection{Scenario B: Periodic Signal Analysis and Phase Constraints}

A dedicated evaluation was performed on periodic signals (sinusoidal and square waves) to evaluate the engine's boundary-condition logic. Commands like \texttt{freq}, \texttt{period}, and periodic \texttt{rms} inherently require a minimum of two full positive zero-crossings to yield valid results. The evaluation verified that the engine successfully measures periodic parameters when sufficient data is available. Crucially, when subjected to a simulation window that covers exactly one period or less, the framework accurately detected the boundary violation and aborted the frequency calculation, preventing the issuance of erroneous infinite or zero frequencies. Furthermore, the engine successfully computed multi-signal metrics, such as the phase \texttt{delay} between two independent square waves.

% ---------------------------------------------------------------------------
\subsection{Scenario C: Expression Parsing Robustness}
\label{sec:calc-verify}

The \texttt{CalcEngine} operates on an Abstract Syntax Tree (AST) constructed via recursive descent parsing. Its robustness was evaluated across four domains:

\begin{itemize}
    \item \textbf{Scalar Arithmetic Stability:} Parameterized test cases verified pure-scalar expressions encompassing varying operator precedence, nested parentheses, and engineering-suffix literals (e.g., \texttt{10n}, \texttt{1p}). The AST evaluator consistently returned values conforming to standard precedence rules with $10^{-9}$ tolerance.
    \item \textbf{Broadcast Semantics and Element-Wise Operations:} The framework's ability to perform vector mathematics was validated by exercising trigonometric (\texttt{sin}, \texttt{cos}, \texttt{tan}), logarithmic (\texttt{ln}, \texttt{log}, \texttt{db}), and multi-signal arithmetic (\texttt{V(1)-V(2)}). Every discrete sample in the resultant vector was cross-verified element-by-element against an expected reference vector, confirming the integrity of the scalar-broadcast and vector-vector alignment logic.
    \item \textbf{Reduction Functions:} Validated that reduction operators (\texttt{max}, \texttt{min}, \texttt{mean}, \texttt{rms}) appropriately collapse dynamic wave vectors into accurate scalars.
    \item \textbf{Mathematical Edge Cases:} The evaluation injected catastrophic mathematical operations, including division by zero (\texttt{1/0}), logarithms of negative numbers, and incomplete syntax (\texttt{V(1)+}). In all cases, the execution halted predictably, flagged the domain error, and preserved the database integrity without system failure.
\end{itemize}

\section{Empirical Performance Evaluation}
\label{sec:pp-performance}

An essential characteristic of an EDA post-processor is its ability to handle millions of data points smoothly. The implemented framework leverages an in-memory caching strategy via the \texttt{WaveformDB}. This section outlines the theoretical performance models and the empirical expectations regarding computational complexity and memory utilization.

\subsection{Time Complexity}

For the primary operations, the asymptotic time cost is strictly linear with respect to the number of waveform samples $N$:

\[
\begin{aligned}
\text{File Parsing (Load)} & \in \mathcal{O}(N) \\
\text{Vector Measurement} & \in \mathcal{O}(N) \\
\text{Expression Evaluation} & \in \mathcal{O}(N) \\
\text{Plot Generation (I/O bounded)} & \in \mathcal{O}(N)
\end{aligned}
\]

By utilizing sequential memory access vectors (\texttt{std::vector}), cache locality is maximized, ensuring that operations such as searching for threshold crossings (\texttt{trise}, \texttt{cross}) or element-wise mathematics execute in fractions of a millisecond for typical circuit sizes.

\subsection{Space Complexity and Trade-offs}

The memory footprint is dominated by the storage of double-precision floating-point samples. The space complexity is modeled as:

\[
\text{Memory Requirement} \approx \mathcal{O}(M \times N)
\]

where $M$ is the number of recorded signals (including currents, voltages, and derived waveforms) and $N$ is the number of time or frequency steps. 

For large-scale integrated circuits (LSI) with tens of thousands of nodes simulated over millions of timesteps, this in-memory strategy introduces a substantial memory trade-off. However, because modern workstations are equipped with significant RAM, trading space for time yields a vastly superior interactive user experience. Users can instantly execute sequential \texttt{meas} and \texttt{calc} commands without enduring repetitive parsing overheads associated with out-of-core processors like traditional SPICE utilities.

\section{Conclusion}
\label{sec:pp-summary}
% ---------------------------------------------------------------------------

In this chapter, a novel, decoupled post-processing framework for electronic simulation analysis was proposed and evaluated. By structurally isolating data acquisition, in-memory waveform management, and specialized computational engines (measurement, expression evaluation, and plotting), the architecture resolves the extensibility and latency limitations of traditional SPICE post-processors. 

Mathematical models underpinning discrete waveform processing were formally defined, and a comprehensive empirical evaluation confirmed the numerical accuracy, robustness against edge cases, and deterministic execution of the framework. The implemented Command-Line Interface provides an interactive REPL suitable for immediate engineering inquiry, alongside powerful batch-scripting support for automated analysis loops. The empirical performance models established that while the in-memory architecture incurs a linear space trade-off, it successfully delivers the sub-millisecond execution latencies required for rapid iterative design.
