\section{Introduction}
\seclbl{sim-intro}

A circuit simulator works like an automated mathematician. It reads a description of your circuit (Chapter~4), converts each component into mathematical equations (Chapter~5), and then solves those equations to predict voltages and currents throughout the circuit. This chapter explains the engine that does the solving.

Figure~\ref{Figure:fig-sim-architecture} shows the full pipeline. The process has four stages:
\begin{enumerate}
  \item \textbf{Setup phase} --- Components from the netlist are registered, devices are instantiated, and the circuit matrices (numerical representations of the equations) are constructed.
  \item \textbf{Analysis dispatch} --- The requested simulation mode (DC bias point, DC sweep, AC frequency sweep, or transient analysis) is selected and configured.
  \item \textbf{Solve phase} --- The nonlinear equations are solved iteratively using a Newton--Raphson loop backed by a specialised sparse linear solver (KLU), with convergence aids for difficult circuits.
  \item \textbf{Output phase} --- Results are written to a standard \texttt{.raw} file compatible with SPICE waveform viewers.
\end{enumerate}

The chapter is organised as follows. Section~6.2 describes the interface between the mathematical modelling layer and the engine. Section~6.3 introduces the four analysis modes in plain terms, explaining what each one does and when to use it. Section~6.4 dives into the linear solver that powers all analysis types. Section~6.5 explains how the engine handles nonlinear devices through the Newton--Raphson method. Section~6.6 provides the detailed transient analysis algorithms. Section~6.7 describes how results are written to output files.

\section{Input from Mathematical Modelling}
\seclbl{engine-input}

Before the engine can run any analysis, it requires two things from the mathematical modelling layer (Chapter~5): the instantiated device objects and the circuit matrices they construct.

The mathematical modelling module delivers:
\begin{itemize}
  \item A vector of \texttt{Device} pointers --- each device knows how to stamp its contributions into the MNA matrices and how to update its nonlinear state during Newton--Raphson iteration.
  \item The \texttt{CircuitMatrix} object, sized according to the node and branch counts determined during device construction. It contains the sparse conductance matrix $\mathbf{G}$ and capacitance matrix $\mathbf{C}$ in compressed-sparse-column (CSC) format.
\end{itemize}

These become the inputs to all analysis modes described below.

\section{Analysis Framework}
\seclbl{analysis-framework}

The engine offers four analysis modes, each suited to a different type of circuit behaviour. Table~\ref{Table:mode-comparison} summarises them at a glance.

The following subsections describe each analysis at a conceptual level; the technical details of how the solver works are covered in Sections~6.4 and~6.5.

\subsection{DC Operating Point (\texttt{.OP})}
\seclbl{analysis-op}

The operating point analysis finds the steady-state voltages and currents in a circuit. Think of it as a snapshot: if you built this circuit on a breadboard and powered it up, what would a multimeter read at each node? Every nonlinear device is active, so the engine must iterate to find the solution --- this is where the Newton--Raphson method (Section~6.5) is essential.

The result serves as the starting point for AC analysis (which linearises transistors around their bias point) and transient analysis (which uses it as the initial condition). When no analysis command is specified, a default operating-point analysis is run automatically. Figure~\ref{Figure:fig-flow-op} illustrates the complete flow, from warm-start through the Newton--Raphson convergence loop to result storage.

\subsection{DC Sweep (\texttt{.DC})}
\seclbl{analysis-dc}

The DC sweep answers the question: ``How does the circuit behave when I vary one source voltage?'' It is like turning a dimmer knob from zero to maximum and watching how the output changes.

The engine sweeps a specified voltage source through a range of values, running an operating-point analysis at each step. Each point starts from the previous point's converged solution, which typically reduces iterations from 5--15 (cold start) to 1--3 per point. The result is a transfer curve --- for example, the input-output characteristic of an amplifier showing gain, linear range, and saturation points. Figure~\ref{Figure:fig-flow-dc} shows the sweep flow, where each point delegates to a full OPAnalysis with warm-start.

\gfigure{DC sweep flow: each sweep point runs a full OPAnalysis, warm-started from the previous point's converged solution.}
  {fig-flow-dc}{width=0.85\textwidth}{Figures/Ch6/flow-dc}

\subsection{AC Small-Signal (\texttt{.AC})}
\seclbl{analysis-ac}

The AC analysis shows how a circuit responds to different frequencies --- like a speaker's frequency response chart or an audio equaliser curve. It sends a small sinusoidal signal through the circuit and measures the output amplitude and phase shift at each frequency.

The engine first finds the DC operating point to linearise all nonlinear devices, then solves the complex system $(\mathbf{G} + j\omega\mathbf{C}) \cdot \mathbf{V} = \mathbf{I}$ at each frequency point (Section~6.4.4). Three sweep types are available: logarithmic decade sweep (\texttt{DEC}), linear sweep (\texttt{LIN}), and logarithmic octave sweep (\texttt{OCT}). The result is a Bode plot showing gain and phase versus frequency. Figure~\ref{Figure:fig-flow-ac} details the AC flow: the DC operating point is only computed when nonlinear devices exist; purely linear circuits skip the OP step and assemble G + C directly.

\gfigure{AC small-signal flow: a nonlinearity check determines whether a DC operating point is needed before assembling the G + C matrices and solving via ComplexKLU at each frequency.}
  {fig-flow-ac}{width=0.85\textwidth}{Figures/Ch6/flow-ac}

\subsection{Transient (\texttt{.TRAN})}
\seclbl{analysis-tran}

Transient analysis shows how voltages and currents evolve over time --- like an oscilloscope recording. You specify a time window, and the engine simulates every moment from start to finish. This is useful for switching behaviour, pulse responses, oscillator startup, and any time-varying signals.

The engine divides time into small steps. At each step it combines the conductance and capacitance matrices with a method-dependent coefficient (Section~6.6), solves the nonlinear system via Newton--Raphson, and estimates the error to decide whether to accept the step or retry with a smaller step. The technical details of timestep control, integration methods, and adaptive sizing are covered in Section~6.6. Figure~\ref{Figure:fig-flow-tran} shows the two-phase transient flow: initial DC bias followed by per-timepoint inline Newton--Raphson.

\section{Linear Solver Architecture}
\seclbl{linear-solver}

Solving a system of thousands of simultaneous equations is the most computationally expensive step in circuit simulation. The engine uses a specialised sparse linear solver designed specifically for circuit matrices --- like having a custom engine in a race car instead of a generic one. The matrices arrive from the circuit assembly layer in CSC format: $\mathbf{G}$ for DC and OP analyses, $\mathbf{G} + j\omega\mathbf{C}$ for AC, and $\mathbf{G} + \alpha\mathbf{C}$ for transient.

\subsection{Abstract Solver Interface}
\seclbl{abstract-solver}

All linear solvers follow a four-phase lifecycle that maps directly to the KLU library API:
\begin{enumerate}
  \item \texttt{analyze()} --- Symbolic analysis: computes a fill-reducing ordering and allocates workspace. Called once when the matrix sparsity pattern is first established.
  \item \texttt{factorize()} --- Numerical LU factorisation: computes $\mathbf{PA} = \mathbf{LU}$. Called on the first Newton--Raphson iteration.
  \item \texttt{refactorize()} --- Numeric refactorisation: updates $\mathbf{L}$ and $\mathbf{U}$ using the existing pivot sequence. Called on subsequent iterations where the sparsity pattern is unchanged, avoiding the expensive reordering step.
  \item \texttt{solve()} --- Forward/backward substitution: $\mathbf{x} = \mathbf{U}^{-1}\mathbf{L}^{-1}\mathbf{P}\mathbf{b}$, writing the solution vector into the right-hand side buffer.
\end{enumerate}

The refactorisation optimisation is critical for performance: it is roughly ten times faster than a full factorisation because it reuses the symbolic analysis. If refactorisation fails (due to numerical instability or pivot tolerance violations), the solver falls back to a full factorisation automatically.

\subsection{KLU Solver}
\seclbl{klu-solver}

KLU (Clark Kent LU), developed by Timothy A.~Davis as part of SuiteSparse, is the standard sparse LU solver for circuit simulation. Circuit MNA matrices are nearly lower-triangular by construction, and KLU exploits this structure through three key configurations:
\begin{itemize}
  \item \textbf{Block Triangular Form (BTF)} --- decomposes the matrix into independent diagonal blocks and factorises each block separately.
  \item \textbf{Approximate Minimum Degree (AMD) ordering} --- reduces fill-in during factorisation.
  \item \textbf{Maximum row scaling} --- improves numerical stability by scaling rows to unit maximum.
\end{itemize}

The solver uses a pivot tolerance of $10^{-3}$ and a minimum pivot threshold of $\text{PIVTOL} = 10^{-13}$ to ensure numerical stability during LU decomposition. Condition number estimation is available through the reciprocal condition estimate, where a value exceeding $10^{12}$ indicates a nearly-singular matrix.

\subsection{Alternative Backend}
\seclbl{nicslu-solver}

NICSLU (Tsinghua University) is a multi-threaded sparse LU solver that provides an alternative backend for large-scale circuits. It uses the same four-phase interface and is selected through conditional compilation.

\subsection{Complex Solver for AC Analysis}
\seclbl{complex-solver}

AC small-signal analysis requires solving the complex system $(\mathbf{G} + j\omega\mathbf{C}) \cdot \mathbf{V} = \mathbf{I}$ at each frequency point. This is handled by a separate complex solver that wraps KLU's complex routines. The matrix is stored in interleaved format: each non-zero entry occupies two consecutive double-precision values representing the real and imaginary parts. The solver supports multiple right-hand sides for multi-port analysis.

\section{Nonlinear Solution --- Newton--Raphson Method}
\seclbl{nr-method}

Transistors, diodes, and other nonlinear devices create equations where the coefficients depend on the solution itself --- like a puzzle where the pieces change shape as you move them. These cannot be solved in one pass. Instead, the engine iterates: linearise around the current guess, solve, check for convergence, and repeat.

\subsection{Newton--Raphson Iteration}
\seclbl{nr-iteration}

Each iteration proceeds as follows:
\begin{enumerate}
  \item Assemble the MNA matrix in DC mode: each device stamps its linearised conductance and current contributions.
  \item Add the minimum conductance ($\text{GMIN} = 10^{-12}$~S) to every diagonal entry for numerical stability.
  \item Finalise the matrix assembly, locking the sparsity pattern on the first iteration and updating numeric values on subsequent iterations.
  \item Solve the linear system using the four-phase KLU lifecycle --- analyse and factorise on the first iteration, refactorise on subsequent iterations.
  \item Apply junction voltage limiting to prevent $\exp(v/V_T)$ overflow in semiconductor models.
  \item Check for convergence using the SPICE criterion (see below).
  \item If converged, store the solution and exit. Otherwise, update the solution vector and repeat.
\end{enumerate}

Up to $\text{ITL1} = 100$ iterations are attempted for DC operating point analysis before falling back to convergence aids. The solution vector is 1-based (index~0 is unused ground) and preserved between calls. This enables warm-starting --- each analysis begins from the previous converged solution, which dramatically reduces iteration counts.

\subsection{SPICE Convergence Criterion}
\seclbl{convergence-criterion}

How close is close enough? The engine checks whether the voltage changed by less than a tiny threshold between successive iterations. For each MNA variable $i$, the change must satisfy:
\[
|V_{\text{new}}[i] - V_{\text{old}}[i]| < \text{TOL} + \text{RELTOL} \cdot \max\left(|V_{\text{new}}[i]|, |V_{\text{old}}[i]|\right)
\]

The parameters controlling convergence are:
\begin{itemize}
  \item $\text{RELTOL} = 10^{-3}$ --- relative error tolerance for all variables.
  \item $\text{VNTOL} = 10^{-6}$~V --- absolute voltage tolerance for node voltages.
  \item $\text{ABSTOL} = 10^{-12}$~A --- absolute current tolerance for branch currents.
\end{itemize}

The convergence checker also detects NaN and Inf values and flags diverging solutions (voltages exceeding $10^6$~V or currents exceeding $10^6$~A).

\subsection{Convergence Aids}
\seclbl{convergence-aids}

When the puzzle refuses to solve, the engine tries easier versions first. Three aids form a fallback chain (Figure~\ref{Figure:fig-fallback-chain}):
\[
\text{Standard NR} \rightarrow \text{GMIN Stepping} \rightarrow \text{Source Stepping} \rightarrow \text{FAIL}
\]

\gfigure{Convergence aids fallback chain.}
  {fig-fallback-chain}{width=0.7\textwidth}{Figures/Ch6/fallback-chain}

\textbf{GMIN stepping} adds a large conductance ($1.0$~S) across every node to ground, making the circuit artificially linear and easy to solve. The conductance is then reduced geometrically over $\texttt{gminSteps} = 10$ stages to the target value ($10^{-12}$~S):
\[
\text{GMIN}_{k} = \frac{\text{GMIN}_{\text{init}}}{r^k}, \quad
r = \left(\frac{\text{GMIN}_{\text{init}}}{\text{GMIN}_{\text{final}}}\right)^{1/\text{steps}}
\]
Each converged step seeds the next, creating a smooth path to the target operating point.

\textbf{Source stepping} ramps all independent source values from zero to their nominal values over $\texttt{sourceSteps} = 10$ configurable steps:
\[
\alpha_k = \frac{k}{\text{numSteps}}, \quad k = 0, 1, \ldots, \text{numSteps}
\]
At each alpha level, the circuit is solved with sources scaled to $\alpha_k$ times their full value.

\textbf{Junction limiting} clamps voltage changes between iterations to a maximum of $3.0$~V per iteration, supplemented by per-device critical-voltage limiting that prevents $\exp(v/V_T)$ overflow in PN junction models.

\subsection{Additional Transient Safeguards}
\seclbl{tran-safeguards}

The transient engine includes two safeguards specific to time-domain simulation. \textbf{Trap ringing detection} monitors for oscillation patterns in the trapezoidal integration method by checking whether consecutive voltage changes have opposite signs. This check is only active when Newton--Raphson requires multiple iterations, preventing false positives on smooth sinusoidal waveforms. An \textbf{LTE reliability guard} skips the local truncation error check when the timestep is within ten times the minimum allowed timestep, because floating-point rounding noise in the second-derivative estimate dominates at extremely small step sizes and can trigger spurious step rejections.

\section{Transient Analysis}
\seclbl{transient-deep}

Transient analysis is the most complex simulation mode, combining the linear solver (Section~6.4), the Newton--Raphson method (Section~6.5), and specialised integration algorithms. While Section~6.3.4 described what transient analysis does, this section explains how it works. The maximum Newton--Raphson iterations per timepoint is $\text{ITL4} = 10$, and the transient error estimation tolerance factor is $\text{TRTOL} = 7.0$.

\subsection{Time-Stepping Loop}
\seclbl{tran-timestepping}

The transient loop proceeds as follows:
\begin{enumerate}
  \item Parse the \texttt{.TRAN} command to obtain timestep parameters.
  \item If \texttt{UIC} (Use Initial Conditions) is not specified, run an initial DC operating point analysis to find the starting bias.
  \item Classify devices into linear and nonlinear subsets.
  \item Create the integration method and timestep controller.
  \item For each timestep $t$ until $t_{\text{stop}}$:
  \begin{enumerate}
    \item Set the transient context (current time, timestep) on all devices.
    \item Update each device's history-dependent internal state.
    \item Assemble the MNA system for transient: combine $\mathbf{G}$ and $\alpha\mathbf{C}$ into a single effective matrix.
    \item Solve the nonlinear system via Newton--Raphson with ITL4 = 10.
    \item Estimate the Local Truncation Error (LTE) and accept or reject the step.
    \item If accepted: advance device states, adjust the next timestep based on LTE, and record output at print intervals.
    \item If rejected: restore the previous state, halve the timestep (minimum 50\% reduction), and retry using backward Euler for numerical damping.
  \end{enumerate}
\end{enumerate}

Breakpoint handling forces the timestep to land exactly on signal transitions (PWL source corners, pulse edges). For three steps following a breakpoint, backward Euler is used to damp numerical oscillations.

\subsection{Integration Methods}
\seclbl{integration-methods}

The capacitor equation $i(t) = C \cdot dv(t)/dt$ is discretised using companion models --- each energy-storage element is replaced by a conductance in parallel with a current source, enabling purely algebraic MNA solution at each timestep:
\[
i(t_{n+1}) = \alpha C \cdot v(t_{n+1}) + i_{\text{history}}
\]

Three implicit methods are available:
\begin{itemize}
  \item \textbf{Backward Euler (BE):} $\alpha = 1/\Delta t$. First-order accurate, highly stable (A-stable), introduces numerical damping. Used as the default for stiff circuits and during post-breakpoint settling.
  \item \textbf{Trapezoidal (TRAP):} $\alpha = 2/\Delta t$. Second-order accurate with better energy conservation. Can exhibit ``trap ringing'' on sharp transitions; this is the default method.
  \item \textbf{Gear-2 (GEAR2):} $\alpha$ depends on both $\Delta t_n$ and $\Delta t_{n-1}$. Second-order with less ringing than trapezoidal.
\end{itemize}

\subsection{Adaptive Timestep Control}
\seclbl{adaptive-timestep}

Just as a video camera records more frames during action scenes and fewer during static shots, the engine automatically adjusts its timestep size based on circuit activity. The timestep controller uses SPICE3-style LTE estimation computed at the device level. Each energy-storage device calculates its LTE using the second derivative of its internal state variable (e.g., charge $q$ for capacitors):
\[
\text{LTE}_q = \frac{\Delta t^2}{12} \cdot \left|\frac{d^2q}{dt^2}\right|
\]

The normalised error per device is:
\[
\epsilon_{\text{dev}} = \frac{|\text{LTE}_q|}{\text{VNTOL} + \text{RELTOL} \cdot \max(|q_n|, |q_{n-1}|)}
\]

The overall error is $\epsilon = \max_i \epsilon_i$. The step is accepted if $\epsilon \leq 1$ and rejected otherwise. The next timestep is computed using the SPICE3 optimal growth formula, scaled by the $\text{TRTOL}$ factor:
\[
\Delta t_{\text{new}} = \Delta t \cdot
\begin{cases}
\min\left(2.0, \; 0.9 \cdot \epsilon^{-1/3}\right), & \epsilon \leq 1 \\[4pt]
\max\left(0.5, \; 0.9 \cdot \epsilon^{-1/3}\right), & \epsilon > 1
\end{cases}
\]

\gfigure{Adaptive timestep during a switching transient.}
  {fig-adaptive-timestep}{width=0.8\textwidth}{Figures/Ch6/fig-adaptive-timestep}

In practice, the timestep shrinks automatically during rapid transitions and grows during quiescent periods, balancing accuracy against simulation time.

\section{Output Generation}
\seclbl{output-generation}

Simulation results are written through a choice of two backends, selected by the \texttt{--raw} command-line flag.

\subsection{Console Output}
\seclbl{console-output}

When no \texttt{--raw} flag is specified, results are printed to the console in plain text. This is used for quick diagnostic runs and automated test validation.

\subsection{Raw File Output}
\seclbl{raw-output}

The primary output format is the NGSpice-compatible \texttt{.raw} file, which any SPICE waveform viewer (GTKWave, ngspice) can open. The file combines an ASCII header with binary double-precision data:
\begin{verbatim}
Title: <circuit_name>
Date: <timestamp>
Plotname: Transient Analysis
Flags: real
No. Variables: 3
No. Points: 1000
Variables:
    0   time        time
    1   V(out)      voltage
    2   I(R1)       current
Binary:
<packed IEEE-754 doubles, row-major>
\end{verbatim}
The \texttt{No. Points:} field is back-patched after all data points are written. For AC analysis, \texttt{flags} is set to \texttt{complex} and each variable occupies two consecutive doubles (real and imaginary parts). Variable types are inferred from header names (\texttt{V(...)} $\rightarrow$ voltage, \texttt{I(...)} $\rightarrow$ current).

\subsection{Diagnostic Reporting}
\seclbl{diag-reporting}

A structured diagnostic system provides status messages for all analysis phases --- OP begin and end, DC sweep per-point progress, AC solver errors, transient step retries, and unhandled commands. All messages are routed through the same output interface, appearing on the console regardless of the output backend.

\section{Summary}
\seclbl{sim-summary}

This chapter presented the simulation engine, the computational core of the SPICE-compatible circuit simulator. The key components are:
\begin{itemize}
  \item An \textbf{interface from mathematical modelling} that delivers device objects and circuit matrices in CSC format.
  \item \textbf{Four analysis modes} --- DC operating point (steady-state snapshot), DC sweep (transfer curves), AC (frequency response), and transient (time-domain evolution).
  \item A \textbf{specialised sparse linear solver} (KLU) with a four-phase lifecycle optimised for circuit matrices, including a refactorisation shortcut that avoids re-ordering on every Newton--Raphson iteration.
  \item A \textbf{Newton--Raphson nonlinear solver} with a three-tier convergence fallback chain (standard NR $\rightarrow$ GMIN stepping $\rightarrow$ source stepping) and additional safeguards for transient simulation.
  \item \textbf{Transient integration} with three implicit methods (backward Euler, trapezoidal, Gear-2) and LTE-based adaptive timestep control.
  \item \textbf{Output generation} that writes NGSpice-compatible raw files through an extensible output interface.
\end{itemize}

The simulation engine seamlessly integrates with the netlist reader (Chapter~4) and the mathematical device models (Chapter~5) to form a complete circuit simulation pipeline. Chapter~7 describes the post-processing tool that reads the raw output files and provides waveform visualisation, measurement, and analysis capabilities.
